# HPL-AI to cuTile Porting Documentation

This document records how the ICL HPL-AI reference implementation was
extracted and ported into the cuTile Python notation in this analysis folder.

Upstream source inspected:

- Repository: <https://bitbucket.org/icl/hpl-ai/src/main/>
- Local inspection commit: `8c7ab180b3278b2b454e6e26da1f22e60463feea`
- Upstream license retained in `hpl_ai/HPL_AI_LICENSE`

The port lives in:

- `hpl_ai/cutile_hpl_ai.py`
- `run_cutile_sweep.sh`
- `setup_cutile_venv.sh`
- `cutile_env.sh`

## Porting Scope

The upstream reference benchmark is a compact C implementation of a
mixed-precision dense linear solve:

1. Generate a matrix and right-hand side.
2. Convert the generated double-precision inputs to lower precision.
3. Factor the matrix with LU without pivoting.
4. Solve the triangular factors.
5. Convert the solution back to higher precision.
6. Run GMRES refinement.
7. Check the HPL-AI scaled residual.

The cuTile port keeps that structure, but separates the GPU-sized workload from
the host refinement:

- cuTile kernels implement conversion, no-pivot LU, and triangular solve.
- Host NumPy implements matrix generation, optional conditioned matrices, GMRES
  refinement, and residual checks.

This is intentional. The extracted cuTile kernels are compact, inspectable, and
useful as NixNan targets. GMRES is much larger and dominated by vector
orthogonalization and small dense solves, so it remains host-side in this first
port.

## Major Design Choices

### Column-Major Layout

The C reference uses `HPLAI_INDEX2D(PTR, R, C, LDIM)`, so `A(i, j)` is stored
as `A[i + j * lda]`. The Python port keeps this layout exactly for the flat
matrix arrays:

```python
row + col * n
```

cuTile kernels operate on the same flat column-major buffer and use
`ct.gather`/`ct.scatter` when a tile lane needs a distinct `(row, col)` element.

### Unblocked LU Instead of Recursive/Blocked LU

The upstream `sgetrf_nopiv.c` uses a blocked wrapper around a recursive
unblocked factorization. The port uses the mathematically equivalent unblocked
outer-product form:

1. Scale the column below the pivot.
2. Apply a rank-1 trailing matrix update.

This maps directly to two simple cuTile kernels per pivot step:

- `scale_column_kernel`
- `rank1_update_kernel`

The numerical operation is still no-pivot LU. The blocking strategy is the
thing that changed.

### Precision Configuration

Precision choices are explicit top-level parameters and are passed through the
major routines via `PrecisionConfig`.

Supported names:

- `float16`
- `float32`
- `float64`

Precision roles:

- `input_precision`: dtype used for generated matrix/RHS storage.
- `factor_precision`: dtype used for LU factorization.
- `solve_precision`: dtype used for the triangular solve RHS/solution buffer.
- `refinement_precision`: dtype used by host GMRES refinement.
- `residual_precision`: dtype used by the scaled residual check and its machine
  epsilon.

The HPL-AI-like default is:

```text
input=float64
factor=float32
solve=float32
refinement=float64
residual=float64
```

### Matrix Conditioning

The upstream HPL-AI matrix generator does not expose a condition-number
parameter. The port keeps that generator as `matrix-kind=hpl-ai` and adds a
second matrix family, `matrix-kind=conditioned`, for sweep studies.

`conditioned` matrices are symmetric positive definite matrices generated by a
seeded QR basis and geometric eigenvalue spectrum. Their 2-norm condition
number is approximately the requested `--condition-number`.

Important:

- `--condition-number` is meaningful for `matrix-kind=conditioned`.
- `matrix-kind=hpl-ai` preserves the upstream row-diagonally-dominant generator
  and reports condition as `n/a`.

## Source-to-Port Mapping

| Upstream C Source | Upstream Functionality | Ported Location |
| --- | --- | --- |
| `matgen.c` | LCG generator, `matgen`, `vecgen` | `_lcg_values`, `hpl_ai_matrix`, `make_rhs` |
| `convert.c` | double/float conversion | CPU `astype`, cuTile `convert_to_f16_kernel`, `convert_to_f32_kernel`, `convert_to_f64_kernel` |
| `sgetrf_nopiv.c` | no-pivot LU | `lu_nopiv_cpu`, `scale_column_kernel`, `rank1_update_kernel` |
| `blas.c` | `strsm`, `dtrsm`, residual helpers | `lu_solve_cpu`, `lu_solve_preconditioner`, triangular cuTile kernels, `scaled_residual` |
| `gmres.c` | Givens rotation and GMRES | `_rotmat`, `_solve_small_upper`, `gmres_refine` |
| `hpl-ai.c` | driver sequence and reporting | `run_cpu`, `run_cutile`, CLI `main` |

## Per-Function Documentation

### `PrecisionConfig`

Groups the five precision choices into one object:

- input
- factor
- solve
- refinement
- residual

All major execution routines accept this configuration rather than hard-coding
HPL-AI's default `float32` factor/solve and `float64` refinement/residual path.

### `MatrixConfig`

Groups matrix-generation parameters:

- `kind`: `hpl-ai` or `conditioned`
- `condition_number`: used by `conditioned`
- `seed`: used by matrix and RHS generation

This keeps matrix size, seed, and conditioning separate from precision choices.

### `SolveResult`

Structured result returned by `run_cpu` and `run_cutile`.

It records:

- backend label
- matrix size and matrix family
- active precision configuration
- elapsed seconds
- solution vector
- scaled residual
- HPL-AI threshold pass/fail
- GMRES iteration count

### `_validate_precision`

Checks that a precision string is one of `float16`, `float32`, or `float64`.
Every dtype helper goes through this validation.

### `_np_dtype`

Maps the precision string to a NumPy dtype. Used by CPU generation,
factorization, solve, GMRES, and residual code.

### `_np_linalg_dtype`

Maps precision strings to a dtype acceptable to NumPy linear algebra routines.
`float16` is promoted to `float32` for the small GMRES Hessenberg solve because
NumPy does not solve `float16` systems directly. The result is cast back to the
requested refinement precision.

### `_cp_dtype`

Maps the precision string to a CuPy dtype. Used only in `run_cutile` when GPU
buffers are allocated.

### `_flat_index`

Implements the HPL-AI column-major address calculation:

```python
row + col * n
```

CPU helpers and documentation use this directly. cuTile kernels inline the same
expression so their AST remains simple.

### `_as_column_major_matrix`

Views a flat 1-D column-major array as a 2-D matrix with `order="F"`. This lets
NumPy code preserve the same layout as the C reference.

### `_lcg_values`

Ports the HPL-AI linear congruential generator from `matgen.c`.

The generator uses:

- multiplier `6364136223846793005`
- increment `1`
- uint64 wraparound
- scale `5.4210108624275222e-20f`
- output range approximately `[-0.5, 0.5)`

The function takes `input_precision` so generated values can be stored as
`float16`, `float32`, or `float64`.

### `hpl_ai_matrix`

Ports `matgen`.

Steps:

1. Generate `n*n` LCG values.
2. Interpret them as a column-major matrix.
3. Compute row absolute sums.
4. Replace the diagonal with `row_sum - abs(original_diagonal)`.

The result is a row-diagonally-dominant matrix matching the reference
construction.

### `conditioned_matrix`

Added by this port for configuration sweeps. This is not in the upstream
reference implementation.

Steps:

1. Create a seeded random matrix.
2. Compute an orthogonal QR basis.
3. Build a geometric eigenvalue spectrum from `condition_number` down to `1`.
4. Form a symmetric positive definite matrix.
5. Return it in flat column-major layout.

This provides an explicit condition-number dimension for testing precision
stress.

### `make_matrix`

Dispatches matrix generation:

- `hpl-ai` -> `hpl_ai_matrix`
- `conditioned` -> `conditioned_matrix`

It passes the top-level `input_precision` into either generator.

### `make_rhs`

Ports `vecgen` using `_lcg_values`. The RHS uses `seed + 1`, matching the
reference driver's pattern.

### `lu_nopiv_cpu`

CPU oracle for no-pivot LU.

It accepts `factor_precision`, casts the matrix to that dtype, and performs the
same unblocked algorithm used by the cuTile kernels:

```text
for k:
  A[k+1:, k] /= A[k, k]
  A[k+1:, k+1:] -= outer(A[k+1:, k], A[k, k+1:])
```

This is used by `run_cpu` and by GMRES as the preconditioner source.

### `lu_solve_cpu`

CPU oracle for the mixed-precision triangular solve.

It accepts `solve_precision`, casts the RHS/solution buffer to that dtype, then
performs:

1. forward solve with unit-lower `L`
2. backward solve with non-unit upper `U`

The LU factors remain in `factor_precision`, so this routine can exercise
mixed factor/solve combinations.

### `lu_solve_preconditioner`

Triangular solve used inside host GMRES.

It accepts `refinement_precision` and casts the LU factors and RHS to that dtype
before applying the same forward/backward solve. This mirrors upstream GMRES,
which applies `U \ (L \ r)` as a left preconditioner.

### `scaled_residual`

Ports the HPL-AI residual check:

```text
||Ax-b||_inf / (eps * (||A||_inf * ||x||_inf + ||b||_inf) * n)
```

The function accepts `residual_precision`. Its epsilon is the machine epsilon
of that dtype divided by two, matching the upstream double-precision formula
when `residual_precision=float64`.

### `_rotmat`

Ports the Givens rotation helper from `gmres.c`.

It computes cosine/sine parameters robustly depending on whether `|b| > |a|`.

### `_solve_small_upper`

Solves the small upper-triangular least-squares problem inside GMRES.

The upstream reference uses `dtrsm` on the small Hessenberg factor. The Python
port uses `np.linalg.solve` and falls back to `np.linalg.lstsq` for singular or
near-singular cases. It accepts `refinement_precision`.

### `gmres_refine`

Ports the reference left-preconditioned GMRES loop.

Major steps:

1. Cast matrix, RHS, and initial solution to `refinement_precision`.
2. Compute preconditioned residual `U \ (L \ (b - A*x))`.
3. Build Krylov vectors with modified Gram-Schmidt.
4. Apply Givens rotations to the Hessenberg matrix.
5. Estimate residual norm from the rotated right-hand side.
6. Update solution when the tolerance is met.
7. Check the scaled residual using `residual_precision`.

The tolerance mirrors the reference:

```text
eps / 2 / (n / 4)
```

where `eps` comes from `refinement_precision`.

### `run_cpu`

Host reference driver.

Sequence:

1. `make_matrix`
2. `make_rhs`
3. `lu_nopiv_cpu`
4. `lu_solve_cpu`
5. optional `gmres_refine`
6. `scaled_residual`
7. return `SolveResult`

This is the executable oracle for the port and lets the sweep script run even
on systems without a CUDA device.

## cuTile Kernels

All cuTile kernels use tile width `W=128`, `ct.bid(0)` for the tile block id,
and `ct.arange(W)` for lane-local offsets.

### `convert_to_f16_kernel`

Loads a tile from the source array, casts it with `ct.astype(..., ct.float16)`,
masks out lanes beyond `total`, and stores to the destination array.

### `convert_to_f32_kernel`

Same as `convert_to_f16_kernel`, but casts to `ct.float32`.

### `convert_to_f64_kernel`

Same as `convert_to_f16_kernel`, but casts to `ct.float64`.

### `scale_column_kernel`

Port of the first half of one no-pivot LU pivot step.

For pivot `k`, each active lane handles one row below the pivot:

```text
A[row, k] = A[row, k] / A[k, k]
```

It uses:

- scalar `ct.load` for pivot
- per-lane `ct.gather` for column entries
- per-lane `ct.scatter` for updated entries

### `rank1_update_kernel`

Port of the Schur complement update in one no-pivot LU pivot step.

Each active lane maps a flattened trailing-matrix offset to:

```text
row = k + 1 + flat % trailing
col = k + 1 + flat // trailing
```

Then it applies:

```text
A[row, col] -= A[row, k] * A[k, col]
```

This is the cuTile form of the upstream `sgemm` trailing update.

### `forward_subtract_kernel`

Port of the lower-triangular solve update.

For pivot `k`, each active lane updates one row below the pivot:

```text
x[row] -= x[k] * L[row, k]
```

This corresponds to the lower `strsm` call in the reference driver.

### `scale_rhs_kernel`

Port of the diagonal division in the upper-triangular solve:

```text
x[k] = x[k] / U[k, k]
```

It runs as a one-block scalar tile kernel.

### `backward_subtract_kernel`

Port of the upper-triangular solve update.

For pivot `k`, each active lane updates one row above the pivot:

```text
x[row] -= x[k] * U[row, k]
```

This corresponds to the upper `strsm` call in the reference driver.

## cuTile Driver Functions

### `_conversion_kernel`

Selects one of the three conversion kernels based on a precision string.

This is how `run_cutile` turns top-level precision parameters into concrete
cuTile conversion calls.

### `_grid_for`

Computes a 1-D tile grid:

```python
((total + W - 1) // W, 1, 1)
```

### `run_cutile`

cuTile driver.

Sequence:

1. Generate matrix/RHS on host with `make_matrix` and `make_rhs`.
2. Allocate CuPy buffers using top-level precision choices.
3. Convert matrix to `factor_precision` with a cuTile conversion kernel.
4. Convert RHS to `solve_precision`.
5. Launch `scale_column_kernel` and `rank1_update_kernel` for each pivot.
6. Launch forward-solve kernels.
7. Launch backward-solve kernels.
8. Convert result to `refinement_precision`.
9. Optionally run host GMRES.
10. Compute scaled residual.

The cuTile backend therefore exercises precision choices in GPU storage and GPU
arithmetic for the extracted LU/solve path.

### `parse_args`

CLI surface for every configurable dimension:

- backend: `cpu` or `cutile`
- matrix size
- matrix family
- condition number
- max GMRES iterations
- seed
- input/factor/solve/refinement/residual precision
- GMRES enable/disable
- optional CPU comparison for cuTile runs

### `_print_result`

Prints one key/value block per run. The sweep script relies on this stable,
line-oriented output.

### `main`

Builds `PrecisionConfig` and `MatrixConfig`, clamps `max_iter` to `n-1`, runs
the selected backend, prints the result, and optionally reports max absolute
difference versus the CPU oracle.

## Runtime Environment

The machine has an RTX 3090 and a loaded NVIDIA 580.126.09 kernel module, but
the default system user-space NVIDIA libraries are 580.159.03. That mismatch
breaks `nvidia-smi` and causes CuPy/cuTile startup failures unless the process
uses the matching 580.126 user-space libraries.

The analysis includes a local environment setup:

```bash
./setup_cutile_venv.sh
source ./cutile_env.sh
```

`setup_cutile_venv.sh` creates `.venv-cutile` with:

- `cuda-tile[tileiras]==1.3.0`
- `cupy-cuda13x==14.0.1`
- `numpy<2.3`

`cutile_env.sh` exports:

- `PYTHON_BIN=.venv-cutile/bin/python`
- `LD_LIBRARY_PATH` with `.cutile-libs`, `/home/ganesh/opt/nv580.126`, and
  CUDA 13.2 toolkit libraries first

It also creates `.cutile-libs` symlinks for the matching `libcuda`,
`libnvidia-ml`, and `libnvidia-ptxjitcompiler` libraries. The local 580.126
library tree is missing `libnvidia-gpucomp.so.580.126.09`, so the script maps
that soname to the available system `libnvidia-gpucomp.so.580.159.03`. This was
enough for cuTile kernel launch on the RTX 3090 in this workspace.

With the environment sourced, this command succeeded:

```bash
python hpl_ai/cutile_hpl_ai.py --backend cutile --n 8 --compare-cpu
```

and reported:

```text

n=8
gmres_iterations=2
scaled_residual=3.539529e-02
passes_hpl_ai_threshold=true
max_abs_diff_vs_cpu=0.000000e+00
```

`run_cutile_sweep.sh` is a top-level bash runner for multiple configurations.
It sources `cutile_env.sh` automatically when present.

Default behavior targets cuTile:

```bash
./run_cutile_sweep.sh
```

Environment variables control the sweep:

| Variable | Default | Meaning |
| --- | --- | --- |
| `PYTHON_BIN` | `python` | Python interpreter |
| `BACKEND` | `cutile` | `cutile` or `cpu` |
| `MATRIX_SIZES` | `16 32 64` | Matrix sizes |
| `MATRIX_KINDS` | `hpl-ai conditioned` | Matrix families |
| `CONDITION_NUMBERS` | `10 1000 100000` | Used for `conditioned` only |
| `INPUT_PRECISIONS` | `float64` | Generated input dtype |
| `FACTOR_PRECISIONS` | `float32` | LU dtype |
| `SOLVE_PRECISIONS` | `float32` | Triangular solve RHS dtype |
| `REFINEMENT_PRECISIONS` | `float64` | GMRES dtype |
| `RESIDUAL_PRECISIONS` | `float64` | Residual-check dtype |
| `MAX_ITER` | `50` | GMRES iteration cap |
| `SEED` | `1` | Matrix/RHS seed |
| `GMRES` | `1` | Set `0` to pass `--no-gmres` |
| `COMPARE_CPU` | `0` | Set `1` to compare cuTile output with CPU |

Example CPU smoke run:

```bash
BACKEND=cpu MATRIX_SIZES="4" MATRIX_KINDS="hpl-ai conditioned" \
  CONDITION_NUMBERS="10" ./run_cutile_sweep.sh
```

Example precision sweep:

```bash
BACKEND=cutile MATRIX_SIZES="32 64" MATRIX_KINDS="conditioned" \
  CONDITION_NUMBERS="10 1000 100000" \
  FACTOR_PRECISIONS="float16 float32" \
```

## Known Limitations

- The cuTile LU uses the unblocked outer-product form rather than the upstream
  recursive/blocked schedule.
- Host GMRES is implemented with NumPy, not cuTile.
- `conditioned` matrices are an added testing feature, not an upstream HPL-AI
  input mode.
- `float16` factorization can overflow or become singular for difficult
  matrices; that is expected and useful for precision stress tests.
